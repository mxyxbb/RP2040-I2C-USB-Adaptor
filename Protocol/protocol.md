## 1. 概述
本协议用于上位机与基于 RP2040 的 USB 转 I2C 适配器之间的串行数据通信。物理层基于 USB CDC 虚拟串口，传输纯字节流。协议采用一问一答的主从模式，上位机发起请求（Request），设备端返回响应（Response）。  
协议支持最高 512 字节的大容量连续读写，并包含双重校验机制（累加和 + 固定帧尾）以保障通信可靠性。  
RP2040固件内包含100ms的超时状态机，以保障在复杂工况下的通信可靠性。  
## 2. 通用帧格式
所有上下行数据帧均采用以下固定的 7 段式结构：  
多字节数据（如长度、波特率）均采用小端模式 (Little-Endian，低字节在前)。  

|偏移量|	字段|	长度 (Byte)|	描述|
|----|----|----|----|
|0|	帧头 (Header)|	1|	固定起始符，值为 0xAA|
|1|	命令字 (CMD)|	1|	区分不同操作。设备响应命令固定为：请求命令 \| 0x80|
|2|	数据长度低字节 (Len_LSB)|	1|	数据域字节总数的低 8 位|
|3|	数据长度高字节 (Len_MSB)|	1|	数据域字节总数的高 8 位|
|4|	数据域 (Payload)|	N|	具体指令的参数或返回数据内容。最大支持 N=512|
|4+N|	校验和 (Checksum)|	1|	8位无符号累加和 (详见第5节)|
|5+N|	帧尾 (Tail)|	1|	固定结束符，值为 0x55|

## 3. 指令集详解
0x01 / 0x81: 获取设备特征码 (Get Signature)  
0x02 / 0x82: 配置 I2C 波特率 (Set Baudrate)  
0x03 / 0x83: I2C 写操作 (I2C Write)  
0x04 / 0x84: I2C 读操作 (I2C Read)  
0x05 / 0x85: 软复位外设 (I2C Reset)  
0x06 / 0x86: I2C 复合写读操作 (Write-Read)  
### 3.1 获取设备特征码 (Get Signature)
用于上位机枚举和识别合法的适配器硬件。  
- 请求 (Host -> Device)
  - CMD: 0x01
  - Len: 0x00
  - Payload: 无
- 正常响应 (Device -> Host)
  - CMD: 0x81
  - Len: N (字符串长度)
  - Payload: 设备的标识字符串 ASCII 码 (如 "RP2040_I2C_ADAPTER_V1")
### 3.2 配置 I2C 波特率 (Set Baudrate)
配置 I2C 总线的工作频率。设备默认波特率为 100,000 Hz。  
- 请求 (Host -> Device)
  - CMD: 0x02
  - Len: 0x0004
  - Payload: 4字节无符号整数 (uint32_t)，表示目标频率(Hz)。采用小端模式 (Little-Endian)。
- 正常响应 (Device -> Host)
  - CMD: 0x82
  - Len: 0x01
  - Payload: 0x00 (表示配置成功)
### 3.3 I2C 写操作 (I2C Write)
向指定 I2C 从机设备写入 N 个字节数据，操作结束后释放总线 (发送 STOP 信号)。  
- 请求 (Host -> Device)
  - CMD: 0x03
  - Len: W + 1 (W 为要写入的数据字节数)
  - Payload:
    - Byte 0: 目标 I2C 从机地址 (7-bit 格式)
    - Byte 1 ~ W: 待写入的数据流
- 正常响应 (Device -> Host)
  - CMD: 0x83
  - Len: 0x0001
  - Payload: 0x00 (表示写入成功，从机有应答)
### 3.4 I2C 读操作 (I2C Read)
从指定 I2C 从机设备直接读取 N 个字节数据。  
- 请求 (Host -> Device)
  - CMD: 0x04
  - Len: 0x0003
  - Payload:
    - Byte 0: 目标 I2C 从机地址 (7-bit 格式)
    - Byte 1 ~ 2: 需要读取的字节数 (小端模式)
- 正常响应 (Device -> Host)
  - CMD: 0x84
  - Len: N (实际读取到的字节数)
  - Payload: 读取到的具体数据流
### 3.5 软复位外设 (I2C Reset)
强制复位并重新初始化 RP2040 的 I2C IP 核，恢复至上一配置的波特率。可以在初始化，或检测到 SCL/SDA 总线死锁或电平异常时使用。  
- 请求 (Host -> Device)
  - CMD: 0x05
  - Len: 0x0000
  - Payload: 无
- 正常响应 (Device -> Host)
  - CMD: 0x85
  - Len: 0x0001
  - Payload: 0x00 (表示复位成功)
### 3.6 I2C 复合写读操作 (Write-Read with Repeated Start)
标准的寄存器读取时序。先写入寄存器地址（不发送 STOP 信号，保持总线），触发 Repeated Start 后，立即读取指定长度的数据。  
- 请求 (Host -> Device)
  - CMD: 0x06
  - Len: W + 3 (W 为要写入的字节数)
  - Payload:
    - Byte 0: 目标 I2C 从机地址 (7-bit 格式)
    - Byte 1 ~ 2: 需要读取的字节数 (R)，16位小端模式
    - Byte 3 ~ (W+2): 待写入的数据（通常为寄存器地址）
- 正常响应 (Device -> Host)
  - CMD: 0x86
  - Len: R (实际读取到的字节数)
  - Payload: 读取到的具体数据流

## 4. 异常与错误处理
当设备执行 I2C 操作失败（如：寻址失败无 ACK 应答、总线仲裁丢失、总线超时等）时，设备会返回一个数据长度 (Len) 为 0 的响应包。  
错误响应格式 (Device -> Host)：  
- CMD: 请求命令 | 0x80
- Len: 0x0000
- Payload: 无
- Checksum: CMD + 0x0000  
上位机在接收到响应帧时，若发现对应操作（CMD 0x83, 0x84, 0x86）的长度为 0，应在界面上提示用户“通信失败”或“无设备应答”。  
错误响应示例： AA 83 00 00 83 55 (针对写操作失败的响应)

## 5. 校验机制说明 (Checksum Mechanism)
为了兼顾微控制器的计算效率和通信的可靠性，本协议采用单字节无符号累加和 (8-bit Unsigned Sum) 算法。  
### 5.1 计算规则
- 计算范围： 仅包含 CMD、Len_LSB、Len_MSB 以及 Payload 的所有字节。
- 排除范围： 不包含 帧头 (0xAA)、校验/自身 以及 帧尾 (0x55)。
- 溢出处理： 将计算范围内的所有字节按无符号 8 位整数 (uint8_t) 进行累加，自然丢弃溢出的高位（即对 256 取模），保留最低 8 位作为最终的校验和。

      #include <stdint.h>
      
      uint8_t CalculateRPI2CChecksum(uint8_t cmd, uint16_t len, const uint8_t* payload) {
          uint8_t len_lsb = len & 0xFF;
          uint8_t len_msb = (len >> 8) & 0xFF;
          uint8_t checksum = cmd + len_lsb + len_msb; 
          
          if (len > 0 && payload != nullptr) {
              for (int i = 0; i < len; ++i) {
                  checksum += payload[i];
              }
          }
          return checksum;
      }

### 5.2 计算实例推演
以 “I2C 复合写读操作” 的测试帧为例： 上位机下发帧：AA 06 03 00 57 01 FF 60 55  
1. 提取参与计算的字节： CMD = 0x06, Len_LSB = 0x03, Len_MSB = 0x00, Payload = [0x57, 0x01, 0xFF]  
2. 执行十进制/十六进制累加：  
3. 0x06 + 0x00 + 0x03 + 0x57 + 0x01 + 0xFF   
= 6 + 3 + 87 + 1 + 255  
= 352 (十进制)   
= 0x0160 (十六进制)  
4. 截断取低 8 位： 0x0160 保留低 8 位为 0x60。  
5. 比对结果： 计算结果 0x60 与帧内校验和字段一致，校验通过。

## 6. 典型指令测试帧示例 (Hex 格式)
可以使用串口调试助手直接发送以下 Hex 字符串进行底层硬件验证。  
### 示例 1：获取设备特征码  
- 发送帧: AA 01 00 00 01 55
  - 解析: AA(包头) 01(命令) 00 00(长度0) 01(校验和=01+00) 55(包尾)
- 预期响应: RP2040 返回 AA 81 15 00 52 50 32 30 34 30 5F 49 32 43 5F 41 44 41 50 54 45 52 5F 56 31 61 55 (包含设备字符串)
### 示例 2：配置波特率至 400kHz
- 发送帧: AA 02 04 00 80 1A 06 00 A6 55
  - 解析: 02(命令) 04 00(长度4) 80 1A 06 00(400000的小端Hex) A6(校验和) 55(包尾)
- 预期响应: AA 82 01 00 83 55 (00代表成功)
### 示例 3：I2C 写操作 (例如向 0x60 地址的从机写入 0x09 寄存器值 0x02)
- 发送帧: AA 03 03 00 60 09 02 71 55
  - 解析: 03(命令) 03 00(长度3) 60(设备地址) 09(寄存器地址) 02(要写入的数据) 71(校验和) 55(包尾)
- 预期响应: AA 83 01 00 00 84 55 (00代表成功)
### 示例 4：I2C 复合写读操作 (例如从 0x60 地址读取 0xFF 寄存器的 1 字节数据)
- 发送帧: AA 06 04 00 60 01 00 FF 6a 55
  - 解析: 06(命令) 04 00(长度4) 60(设备地址) 01 000(读取1字节) FF(目标寄存器) 6a(校验和) 55(包尾)
- 预期响应: AA 86 01 00 00 87 55 (假设读到的数据为 0x00)
### 示例 5：发送软复位指令
- 发送帧: AA 05 00 00 05 55
  - 解析: 05(命令) 00 00(长度) 05(校验和) 55(包尾)
- 预期响应: AA 85 01 00 00 86 55
